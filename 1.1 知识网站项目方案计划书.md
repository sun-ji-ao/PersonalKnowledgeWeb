# 个人知识网站项目方案计划书

## 1. 项目目标

基于仓库内 `bypassAV-study-main` 目录中已有的 Markdown 技术文档，搭建一个低成本、易维护、可持续扩展的个人知识网站。网站初期以“结构化技术文档浏览”为核心，当前已演进为具备首页博客信息流、分类、标签、归档和全文搜索的个人技术知识站。

核心要求：

- 保留现有 Markdown 文件的层级关系。
- 原生支持 Markdown、代码高亮、文章目录导航。
- 支持分类、标签、站内全文搜索。
- UI 风格极简、科技感、阅读体验优秀。
- 部署成本低，优先使用免费方案。
- 后续可持续新增、修改、发布文章。

## 2. 当前内容结构判断

目标目录下现有约 251 个 Markdown 文件，现有内容已经具备较清晰的课程专题层级，例如：

- `01_C&C++快速入门`
- `02_x86_64架构汇编语言`
- `03_C&C++软件逆向技术`
- `04_Win32编程`
- `05_进程通信专题`
- `06_Windows内核安全`
- `07_Windows网络编程`
- `08_Windows_PE结构`
- `09_加密壳`
- `10_shellcode开发技术`
- `11_shellcode_loader`
- `20_免杀专题`

这些目录天然适合映射为网站的一级分类。目录下的文章文件支持两种命名方式：

| 文件名格式 | 示例 | 页面展示 |
| --- | --- | --- |
| `课时XX_标题.md` | `课时01_Windows环境配置.md` | `1、Windows环境配置` |
| `XX_标题.md` | `08_Windows 全架构 Hook 技术图谱-用户态 Hook.md` | `8、Windows 全架构 Hook 技术图谱-用户态 Hook` |

建议初期不要强行重写所有 Markdown，而是先通过内容导入脚本做“轻量规范化”：

- 保留原始目录名，作为分类来源。
- 从文件名中提取编号和标题。
- 自动生成或补齐 frontmatter。
- 将原始 Markdown 转换到站点内容目录。
- 后续再逐步人工优化标题、摘要、标签和封面。

## 3. 推荐技术选型

### 3.1 当前采用方案：Astro + Pagefind

当前实际采用：

- Astro：静态站点生成器，构建速度快，部署简单。
- 自定义 Astro 页面与组件：实现首页双栏博客流、文章页和归档页。
- Shiki：代码高亮，Astro 原生集成。
- Pagefind：静态全文搜索，不依赖后端服务。
- GitHub Pages / Cloudflare Pages：免费部署。
- Cloudflare DNS：绑定自己的域名，支持 HTTPS。

说明：

- 方案初期曾考虑 Starlight，但当前版本未接入 Starlight，而是通过自定义页面实现更灵活的博客化首页和文章页布局。

### 3.2 备选方案对比

| 方案 | 优点 | 不足 | 适用场景 |
| --- | --- | --- | --- |
| Astro + Starlight | 文档体验强、部署简单、性能好 | 博客标签体系需要少量扩展 | 本项目首选 |
| Docusaurus | 文档 + 博客体系成熟 | React 生态稍重，主题风格偏固定 | 需要完整博客系统 |
| VitePress | 简洁轻量，适合文档站 | 博客、标签、搜索需要自行扩展 | 偏技术文档 |
| Hugo | 极快，主题多 | 中文技术内容和前端扩展体验一般 | 纯博客优先 |
| Next.js | 灵活度最高 | 需要写更多代码，维护成本高 | 后期做复杂 Web 应用 |

综合判断：本项目当前实际落地为 `Astro + Pagefind + 自定义页面`，未使用 Starlight。

## 5. 核心功能设计

### 5.1 Markdown 原生支持

初期支持：

- 标题、列表、引用、表格。
- 代码块和行内代码。
- 图片、链接。
- 目录导航。
- Frontmatter 元数据。

后续增强：

- Mermaid 图表。
- 数学公式。
- 提示块，例如 Note、Tip、Warning。
- 文件树展示组件。
- 代码文件引用组件。

### 5.2 代码高亮

使用 Shiki 作为代码高亮引擎。

建议默认支持语言：

- C
- C++
- C#
- Go
- Python
- PowerShell
- Bash
- Assembly
- JavaScript
- TypeScript
- JSON
- YAML

推荐主题：

- 暗色：`github-dark` 或 `one-dark-pro`
- 亮色：`github-light`

站点默认建议使用暗色科技风，同时保留明暗主题切换。

### 5.3 分类体系

分类直接来自一级目录，展示顺序按目录名前缀序号排序，例如 `01_`、`02_`、`20_`。

分类页、首页信息流和文章详情页中的专题名均可点击，跳转到 `/categories/专题名/`。

| 源目录 | 网站分类 |
| --- | --- |
| `01_C&C++快速入门` | C&C++快速入门 |
| `02_x86_64架构汇编语言` | x86 64架构汇编语言 |
| `04_Win32编程` | Win32编程 |
| `06_Windows内核安全` | Windows内核安全 |
| `08_Windows_PE结构` | Windows PE结构 |
| `15_HOOK专题` | HOOK专题 |

分类字段建议写入 Markdown frontmatter：

```yaml
category: Windows 内核安全
```

### 5.4 标签体系

标签不建议完全依赖目录，应允许一篇文章拥有多个标签。

初期可以自动推断：

- `C++`
- `Win32`
- `x86`
- `x64`
- `PE`
- `Windows Kernel`
- `Driver`
- `Network`
- `IPC`
- `Reverse Engineering`

后续人工维护：

```yaml
tags:
  - Windows
  - Kernel
  - Driver
```

### 5.5 站内全文搜索

使用 Pagefind。

优点：

- 不需要后端服务。
- 构建时生成搜索索引。
- 支持中文搜索。
- 可部署在 GitHub Pages、Cloudflare Pages、Vercel 等静态平台。
- 不需要 Algolia 账号或额外付费服务。

### 5.6 文章目录导航

每篇文章左侧栏显示当前文章目录，来源于 Markdown 标题层级。

当前实现：

- 文章目录位于左侧栏分类和标签信息下方。
- 导入脚本会为未编号标题补齐 `1、`、`1.1、` 形式编号。
- 正文标题与目录编号保持一致。

建议规范：

- 页面只使用一个一级标题。
- 正文章节从二级标题开始。
- 标题层级不要跳级。
- 长文必须有清晰的小节。

### 5.7 博客化首页

当前已实现博客化首页：

- 左侧科技风导航栏。
- 右侧文章信息流。
- 每页 10 条，按展示日期倒序。
- 支持分页浏览。
- 支持分类、标签、归档、知识库和 RSS 入口。
- 文章标题统一显示为 `编号、文章名`。
- 信息流中的专题名可点击，跳转到对应分类页。

最终形态：

```text

├─ 左侧导航
├─ 文章信息流
├─ 分页
└─ 分类入口

知识库
├─ C/C++
├─ 汇编
├─ Win32
├─ 内核安全
├─ PE
└─ 网络编程
```

## 7. 部署方案

### 7.1 推荐部署：GitHub Pages + 阿里云 DNS

当前推荐方案：

- 部署平台：GitHub Pages。
- 自动部署：GitHub Actions。
- 自定义域名：`sunjiao.me`（阿里云已购买）。
- DNS 服务商：阿里云云解析 DNS。

详细操作见 `1.5 部署上线操作手册.md`。

部署流程摘要：

1. 推送代码到 `sun-ji-ao/PersonalKnowledgeWeb`。
2. 启用 GitHub Pages（Source 选择 GitHub Actions）。
3. 由 `.github/workflows/deploy.yml` 自动执行 `npm run build` 并发布 `dist/`。
4. 在 GitHub Pages 中配置自定义域名 `sunjiao.me`。
5. 在阿里云 DNS 添加 4 条 A 记录和 `www` CNAME。
6. 启用 HTTPS 并验证首页、RSS、Sitemap。

### 7.2 备选部署：Cloudflare Pages

推荐原因：

- 免费额度足够个人站点使用。
- 国内外访问稳定性通常较好。
- 自定义域名和 HTTPS 配置简单。
- 可直接连接 GitHub 仓库自动部署。
- 后续可接入 Cloudflare Analytics、Turnstile、Access 等能力。

部署流程：

1. 创建 GitHub 仓库，例如 `personal-knowledge-site`。
2. 将 Astro 项目推送到 GitHub。
3. 登录 Cloudflare Pages。
4. 连接 GitHub 仓库。
5. 设置构建命令：`npm run build`。
6. 设置输出目录：`dist`。
7. 绑定自有域名。
8. 在 DNS 中添加 CNAME 或按 Cloudflare 指引托管域名。

### 7.3 备选部署：GitHub Pages 默认域名

优点：

- 完全免费。
- 与 GitHub 仓库集成简单。
- 适合纯静态站点。

不足：

- 国内访问速度不一定稳定。
- 自定义域名和 HTTPS 可用，但整体体验不如 Cloudflare Pages。

### 7.4 域名建议

当前已购买域名：

```text
sunjiao.me
```

推荐配置：

```text
主站：sunjiao.me
```

## 8. 内容维护方案

### 8.1 推荐工作流

```text
本地编辑 Markdown
        |
运行导入脚本
        |
本地预览
        |
提交 Git
        |
GitHub 自动触发部署
        |

```

常用命令建议：

```bash
npm run import

npm run build
git add .
git commit -m "更新知识库文章"
git push
```

### 8.2 文章 frontmatter 规范

建议所有文章统一使用：

```yaml
---
title: 第一个驱动程序
description: Windows 内核驱动入门实验记录
category: Windows 内核安全
tags:
  - Windows
  - Kernel
  - Driver
date: 2026-07-14
updated: 2026-07-14
draft: false
order: 2
useRealDate: true
---
```

字段说明：

| 字段 | 用途 |
| --- | --- |
| `title` | 页面标题 |
| `description` | SEO 和列表摘要 |
| `category` | 文章分类 |
| `tags` | 标签 |
| `date` | 创建日期 |
| `updated` | 更新日期 |
| `draft` | 是否草稿 |
| `order` | 同分类下排序 |
| `useRealDate` | 是否使用真实上传日期展示 |

### 8.3 展示日期规则

当前采用双轨日期策略：

| 文章类型 | 判断方式 | 展示日期 |
| --- | --- | --- |
| 历史文章 | `content_dates.json` 中 `useRealDate: false` | 随机分布在 2021-11-01 至 2025-10-31 |
| 2026年5月后新增 | 首次导入时 `useRealDate: true` | 使用真实上传日期 |

说明：

- `content_dates.json` 会持久化每篇文章的日期记录，需纳入版本控制。
- 首次导入时，现有文章会自动登记为历史文章。
- 之后在 `bypassAV-study-main` 中新增的 `.md` 文件，导入时会自动记录真实日期。

## 10. 风险与注意事项

### 10.1 内容合规风险

当前内容包含逆向、安全、内核、Shellcode 等高敏感技术主题。若网站公开访问，建议：

- 明确站点定位为安全研究、防御学习和技术笔记。
- 避免发布可直接用于未授权攻击的操作化内容。
- 对高敏感文章设置草稿、私有仓库或访问控制。
- 可使用 Cloudflare Access 为部分路径增加登录保护。

### 10.2 Markdown 兼容风险

不同来源的 Markdown 可能存在：

- 图片路径失效。
- 标题层级混乱。
- 代码块未标注语言。
- 表格格式不规范。
- 文件名包含特殊字符。

解决策略：

- 第一轮只做无损导入。
- 第二轮自动修复明显格式问题。
- 第三轮人工优化重点文章。

### 10.3 中文搜索风险

静态搜索对中文分词支持可能不如后端搜索引擎。

解决策略：

- 初期使用 Pagefind，成本最低。
- 文章标题、摘要、标签尽量补充关键词。
- 后续如搜索需求增强，可接入 Meilisearch、Typesense 或 Algolia。
