# 个人知识网站项目方案计划书

## 1. 项目目标

基于本地目录 `C:\Users\17536\Desktop\bypassAV-study-main` 中已有的 Markdown 技术文档，搭建一个低成本、易维护、可持续扩展的个人知识网站。网站初期以“结构化技术文档浏览”为核心，后续逐步演进为个人技术博客。

核心要求：

- 保留现有 Markdown 文件的层级关系。
- 原生支持 Markdown、代码高亮、文章目录导航。
- 支持分类、标签、站内全文搜索。
- UI 风格极简、科技感、阅读体验优秀。
- 部署成本低，优先使用免费方案。
- 后续可持续新增、修改、发布文章。

## 3. 推荐技术选型

### 3.1 首选方案：Astro + Starlight + Pagefind

推荐使用：

- Astro：静态站点生成器，构建速度快，部署简单。
- Starlight：Astro 官方文档主题，适合知识库、技术文档和课程内容。
- Shiki：代码高亮，Astro/Starlight 原生集成。
- Pagefind：静态全文搜索，不依赖后端服务。
- GitHub Pages / Cloudflare Pages：免费部署。
- Cloudflare DNS：绑定自己的域名，支持 HTTPS。

选择原因：

- 不需要服务器，最终产物是静态文件。
- 免费部署即可满足个人博客和知识库需求。
- Markdown 支持成熟，代码块高亮效果好。
- 文档目录、侧边栏、页面目录导航开箱即用。
- 后续可扩展博客、标签、RSS、站点地图、阅读统计等能力。

### 3.2 备选方案对比

| 方案 | 优点 | 不足 | 适用场景 |
| --- | --- | --- | --- |
| Astro + Starlight | 文档体验强、部署简单、性能好 | 博客标签体系需要少量扩展 | 本项目首选 |
| Docusaurus | 文档 + 博客体系成熟 | React 生态稍重，主题风格偏固定 | 需要完整博客系统 |
| VitePress | 简洁轻量，适合文档站 | 博客、标签、搜索需要自行扩展 | 偏技术文档 |
| Hugo | 极快，主题多 | 中文技术内容和前端扩展体验一般 | 纯博客优先 |
| Next.js | 灵活度最高 | 需要写更多代码，维护成本高 | 后期做复杂 Web 应用 |

综合判断：本项目应优先使用 `Astro + Starlight + Pagefind`。

```text

```

## 5. 核心功能设计

### 5.1 Markdown 原生支持

初期支持：

- 标题、列表、引用、表格。
- 代码块和行内代码。
- 图片、链接。
- 目录导航。
- Frontmatter 元数据。

后续增强：

- Mermaid 图表。
- 数学公式。
- 提示块，例如 Note、Tip、Warning。
- 文件树展示组件。
- 代码文件引用组件。

### 5.2 代码高亮

使用 Shiki 作为代码高亮引擎。

建议默认支持语言：

- C
- C++
- C#
- Go
- Python
- PowerShell
- Bash
- Assembly
- JavaScript
- TypeScript
- JSON
- YAML

推荐主题：

- 暗色：`github-dark` 或 `one-dark-pro`
- 亮色：`github-light`

站点默认建议使用暗色科技风，同时保留明暗主题切换。

### 5.3 分类体系

分类直接来自一级目录，例如：

| 源目录 | 网站分类 |
| --- | --- |
| `01_C&C++快速入门` | C/C++ 快速入门 |
| `02_x86_64架构汇编语言` | x86/x64 汇编 |
| `04_Win32编程` | Win32 编程 |
| `06_Windows内核安全` | Windows 内核安全 |
| `08_Windows_PE结构` | Windows PE 结构 |

分类字段建议写入 Markdown frontmatter：

```yaml

```

### 5.4 标签体系

标签不建议完全依赖目录，应允许一篇文章拥有多个标签。

初期可以自动推断：

- `C++`
- `Win32`
- `x86`
- `x64`
- `PE`
- `Windows Kernel`
- `Driver`
- `Network`
- `IPC`
- `Reverse Engineering`

后续人工维护：

```yaml
tags:
  - Windows
  - Kernel
  - Driver
```

### 5.5 站内全文搜索

使用 Pagefind。

优点：

- 不需要后端服务。
- 构建时生成搜索索引。
- 支持中文搜索。
- 可部署在 GitHub Pages、Cloudflare Pages、Vercel 等静态平台。
- 不需要 Algolia 账号或额外付费服务。

### 5.6 文章目录导航

每篇文章右侧显示当前文章目录，来源于 Markdown 标题层级。

建议规范：

- 页面只使用一个一级标题。
- 正文章节从二级标题开始。
- 标题层级不要跳级。
- 长文必须有清晰的小节。

### 5.7 博客化演进

第一阶段以文档站为主，后续增加博客能力：

- 首页展示精选文章。
- 文章归档页。

## 7. 部署方案

### 7.1 首选部署：Cloudflare Pages

推荐原因：

- 免费额度足够个人站点使用。
- 国内外访问稳定性通常较好。
- 自定义域名和 HTTPS 配置简单。
- 可直接连接 GitHub 仓库自动部署。
- 后续可接入 Cloudflare Analytics、Turnstile、Access 等能力。

部署流程：

1. 创建 GitHub 仓库，例如 `personal-knowledge-site`。
2. 将 Astro 项目推送到 GitHub。
3. 登录 Cloudflare Pages。
4. 连接 GitHub 仓库。
5. 设置构建命令：`npm run build`。
6. 设置输出目录：`dist`。
7. 绑定自有域名。
8. 在 DNS 中添加 CNAME 或按 Cloudflare 指引托管域名。

### 7.2 备选部署：GitHub Pages

优点：

- 完全免费。
- 与 GitHub 仓库集成简单。
- 适合纯静态站点。

不足：

- 国内访问速度不一定稳定。
- 自定义域名和 HTTPS 可用，但整体体验不如 Cloudflare Pages。

### 7.3 域名建议

建议设置：

```text
主站：www.yourdomain.com
根域：yourdomain.com
知识库：kb.yourdomain.com
博客：blog.yourdomain.com
```

初期可以只使用：

```text
www.yourdomain.com
```

后续再拆分子域名。

## 10. 风险与注意事项

### 10.1 内容合规风险

当前内容包含逆向、安全、内核、Shellcode 等高敏感技术主题。若网站公开访问，建议：

- 明确站点定位为安全研究、防御学习和技术笔记。
- 避免发布可直接用于未授权攻击的操作化内容。
- 对高敏感文章设置草稿、私有仓库或访问控制。
- 可使用 Cloudflare Access 为部分路径增加登录保护。

### 10.2 Markdown 兼容风险

不同来源的 Markdown 可能存在：

- 图片路径失效。
- 标题层级混乱。
- 代码块未标注语言。
- 表格格式不规范。
- 文件名包含特殊字符。

解决策略：

- 第一轮只做无损导入。
- 第二轮自动修复明显格式问题。
- 第三轮人工优化重点文章。

### 10.3 中文搜索风险

静态搜索对中文分词支持可能不如后端搜索引擎。

解决策略：

- 初期使用 Pagefind，成本最低。
- 文章标题、摘要、标签尽量补充关键词。
- 后续如搜索需求增强，可接入 Meilisearch、Typesense 或 Algolia。
